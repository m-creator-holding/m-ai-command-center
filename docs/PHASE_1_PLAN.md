# План Фазы 1 — «Реальное время и переподключение»

- **Статус:** подготовка (Фаза 0 закрыта; Фаза 1 — после go Владельца по объёму)
- **Дата:** 2026-06-14
- **Контекст:** `CONCEPT_bridge_owner_cockpit.md` §10, `adr/0001..0003`, код `command-center-engine/`

---

## Что Фаза 1 даёт (из концепта §10)

> WSS-канал, поток действий ИИ в чат, кнопки подтверждения опасных действий,
> журнал событий с номерами и догрузкой после обрыва. Здесь же — базовый аудит.

Цель простыми словами: чат работает как живой и надёжный — действия Альфреда
видно в реальном времени, опасное действие требует нажатия «подтвердить», а при
обрыве связи переписка не теряется и догружается с того места, где прервалась.

## Ключевые находки (учесть в реализации)

1. **Сквозная нумерация — ИСПРАВЛЕНО заранее.** `AdapterState` создавался на
   каждый запуск, seq шёл 1,2,3 на каждое сообщение. Догрузка по seq на этом
   была бы сломана. SessionManager теперь перештамповывает сквозным seq
   (коммит f909d12, тест в `test_session_seq.py`).
2. **Журнал аудита = источник для догрузки.** `engine/audit.py` уже пишет все
   события в JSONL с seq. Догрузка после обрыва = чтение журнала и
   воспроизведение событий с seq > last_seq клиента. Отдельное хранилище для
   Фазы 1 не нужно — переиспользуем журнал (схему БД отложить до фазы масштаба).
3. **proposal/apply уже сквозной в движке.** Событие proposal несёт готовый
   `change` для кадра apply — фронту остаётся показать карточку и вернуть apply.
4. **Сеть песочницы пока хостовая** (ADR-0002): WSS/TLS и ограничение доступа —
   часть этой фазы (периметр) либо фазы закалки; для Фазы 1 минимум — TLS перед
   Шлюзом + проверка, что порт Шлюза только на localhost.

## Декомпозиция Фазы 1

### Бэкенд (движок/шлюз)
- **P1-B1. Догрузка из журнала.** Кадр `subscribe` принимает `from_seq`; Шлюз
  читает журнал сессии и воспроизводит события с `seq > from_seq` перед живым
  потоком. DoD: клиент переподключился — не потерял ни одного события.
- **P1-B2. Устойчивость переподключения.** Шлюз переживает разрыв WebSocket без
  падения сессии; статус `session.state` отдаёт актуальный `last_seq`.
- **P1-B3. Подтверждение опасного действия сквозь канал.** Кадр `apply` уже
  есть; добавить путь «ожидающее подтверждение» в статус сессии (чтобы при
  переподключении карточка восстановилась).

### Фронтенд (приложение)
- **P1-F1. Живой чат проекта на WSS** вместо мока для выбранного проекта
  (engineClient уже есть; включить per-project, как для Сайта НЗСН).
- **P1-F2. Карточка подтверждения по событию proposal.** Событие proposal →
  ConfirmCard (она есть) с человекочитаемым описанием правки + кнопки
  «Подтвердить»/«Отклонить»; «Подтвердить» шлёт кадр apply, показывает
  apply.result. Никакого техжаргона.
- **P1-F3. Переподключение без потерь.** Клиент хранит last_seq, при обрыве
  переподключается с `from_seq=last_seq`, дорисовывает пропущенное.

### Инфраструктура
- **P1-I1. WSS перед Шлюзом.** Reverse-proxy (nginx/Caddy) + TLS (Let's Encrypt);
  порт Шлюза только на localhost; наружу — только прокси. Аутентификация —
  минимум токен (полноценный вход — Фаза 2 с биометрией).
- **P1-I2. Туннель/доступ Владельцу** для живого теста с телефона (это же
  закрывает показ Фазы 0 вживую) — отдельное согласие Владельца.

### Качество
- **P1-Q1.** Тесты догрузки (обрыв на N-м событии → без потерь), идемпотентность
  apply, переподключение. Симуляционно ($0), живой прогон — точечно.

## DoD Фазы 1

1. Открыл чат на телефоне/Mac, дал команду на правку → увидел предложение →
   нажал «Подтвердить» → файл изменился → результат в чате. Всё по защищённому
   каналу (WSS).
2. Связь оборвалась и восстановилась → переписка и статус догрузились без
   потерь, по номерам событий.
3. Каждое действие — в журнале аудита.

MVP-напоминание: Фаза 1 + Фаза 2 (Mac-приложение в доке) завершают MVP.

## Не входит в Фазу 1 (чтобы не расползался объём)

  (часть изоляции уже взята заранее, ADR-0002).
