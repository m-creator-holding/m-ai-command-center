## 1. Какие реально есть способы запускать Claude с инструментами на своём сервере

Anthropic на дату сверки предлагает три различимые поверхности (плюс «голый» Messages API, который для нашего кейса нерелевантен — там tool-loop пишешь сам).

### 1.1. Claude Agent SDK (Python / TypeScript) — это и есть «вариант A» архитектора
**[подтверждено]** «The Agent SDK gives you the same tools, agent loop, and context management that power Claude Code, programmable in Python and TypeScript.» Установка: `pip install claude-agent-sdk` (Python ≥3.10) / `npm install @anthropic-ai/claude-agent-sdk`. TS-пакет тащит нативный бинарь Claude Code как опциональную зависимость — отдельно Claude Code ставить не надо.
- **Где исполняется код:** «**Runs in: Your process, your infrastructure**», «Agent works on: **Files on your infrastructure**», «Session state: **JSONL on your filesystem**». То есть руки ИИ — прямо на нашем VPS, в реальных папках. Это ровно то, что нужно концепту.
- **Зрелость:** распространяется как GA-библиотека (публичный pip/npm, ToS — Commercial Terms), beta-заголовков не требует. Явной плашки «GA» в overview нет, но позиционируется как production-инструмент («CI/CD pipelines», «Production automation» — рекомендованный выбор именно SDK). **[подтверждено как production-grade; формального слова "GA" в тексте нет]**

### 1.2. Claude Code в headless / SDK-режиме
**[подтверждено]** Это та же поверхность, что и Agent SDK — Anthropic прямо приравнивает: «Agent SDK vs Claude Code CLI — Same capabilities, different interface». CLI хорош для интерактивной разработки и разовых задач; SDK — для кастомных приложений и production-автоматизации. Для «Мостика» (серверный долгоживущий процесс под управлением нашего gateway) правильная форма — **именно SDK как библиотека**, а не оборачивание CLI. CLI/`claude -p` остаётся как ручной инструмент.

### 1.3. Managed Agents — это «вариант B» архитектора
**[подтверждено]** «Managed Agents is a hosted REST API: Anthropic runs the agent and the sandbox». Концепты: Agent / Environment / Session / Events. Эндпоинты `/v1/agents`, `/v1/sessions` (+ SSE-стрим сессии). **Beta**, заголовок `managed-agents-2026-04-01` (SDK ставит его сам). Включено по умолчанию на всех API-аккаунтах.
- **Важное уточнение против ADR:** в ADR-0001 §Решение-1 и в концепте §4.1 написано, что окружения «только cloud». **Это устарело.** Официально: «Environment — an Anthropic-managed cloud sandbox, **or a self-hosted sandbox on your own infrastructure**». Есть отдельная страница self-hosted sandboxes. Инженерный блог: «Self-hosted sandboxes are now available… as an alternative to running tool execution in Anthropic's infrastructure», и токены изолированы от песочницы («tokens are never reachable from the sandbox where Claude's generated code runs»). **[подтверждено — корректировка факта в ADR]**

## 2. Возможности по каждому движку (то, что важно для «мозга 24/7» и стрима наружу)

| Способность | Agent SDK (вариант A) | Managed Agents (вариант B) |
|---|---|---|
| Долгоживущая многоповоротная сессия в одном процессе | **[подтверждено]** Python `ClaudeSDKClient` — держит сессию, каждый `client.query()` продолжает её без передачи ID; `connect()/disconnect()`. TS — `continue: true` на каждый вызов. | **[подтверждено]** Сессии long-running, «resume cleanly after pauses», состояние на стороне Anthropic |
| Возобновление после рестарта процесса | **[подтверждено]** `resume=<session_id>` или `continue_conversation`. Состояние — JSONL в `~/.claude/projects/<encoded-cwd>/<id>.jsonl`. Есть `list_sessions()`, `get_session_messages()`, `fork_session`, `rename/tag`. | **[подтверждено]** `getSession(id)` возвращает event-log, `wake(sessionId)` поднимает новую сессию при падении |
| Реальные инструменты (файлы / bash) | **[подтверждено]** встроенные Read/Write/Edit/Bash/Glob/Grep/Monitor/WebSearch/WebFetch | **[подтверждено]** Bash + file ops + web + MCP, но в **песочнице** (cloud или self-hosted) |
| Запуск суб-агентов | **[подтверждено]** `agents={...}` через `AgentDefinition` (программно) ИЛИ файлы `.claude/agents/*.md`; вызов через инструмент `Agent`; у дочерних сообщений поле `parent_tool_use_id` — удобно для TraceTree | **[не подтверждено в этой сверке]** прямого упоминания суб-агентов в overview Managed Agents не встретил |
| Стрим событий наружу | **[подтверждено]** async-итератор сообщений; частичные апдейты при `include_partial_messages=True`; хуки `PreToolUse/PostToolUse/Stop/SessionStart/SessionEnd/...` — основа для «живого потока действий» и аудита | **[подтверждено]** стрим по SSE; история событий persist на сервере и доступна целиком (`GET /v1/sessions/{id}/stream`) |
| MCP | **[подтверждено]** `mcp_servers={...}` | **[подтверждено]** MCP-серверы поддержаны |
| Чтение настроек проекта (CLAUDE.md, skills, commands) | **[подтверждено]** грузит `.claude/` из cwd и `~/.claude/`; управляется через `setting_sources`/`settingSources` | n/a (среда другая) |

**Вывод по §2:** Agent SDK закрывает все пять требований концепта (24/7-сессия, многоповорот, реальные руки, суб-агенты, стрим наружу) штатными механизмами. Managed Agents тоже почти всё умеет, но руки — в песочнице, а суб-агенты по этой сверке не подтверждены.

## 3. Изоляция и безопасность — где исполняется код

| | Agent SDK (A) | Managed Agents cloud | Managed Agents self-hosted |
|---|---|---|---|
| Где бежит код | Наш VPS, наш процесс, реальные `/root/<pod>/` | Контейнер Anthropic | Песочница на **нашей** инфраструктуре |
| Доступ к боевым папкам | Прямой | Только смонтированные копии | Своя инфраструктура, ближе к боевой |
| Секреты | Под нашим контролем | Тянуть в чужой контейнер нежелательно | Токены изолированы от песочницы по дизайну |
| Изоляция сессий проектов | **Наша ответственность:** разный `cwd` на сессию (сессии и так привязаны к `<encoded-cwd>`), ограничение `allowed_tools`, хуки `PreToolUse` как «красная зона», отдельные права | Изоляция между сессиями встроена (песочница на сессию) | Песочница на сессию + наш периметр |

**Замечания по изоляции для варианта A (важно для §7 концепта):**
- Сессии физически разделяются по рабочей папке (`cwd`) — это совпадает с правилом холдинга «один pod = своя папка/агенты». **[подтверждено механизмом cwd-привязки сессий]**
- Но Agent SDK сам по себе **не песочница ОС**: процесс с инструментом Bash имеет права пользователя, под которым запущен. Жёсткую изоляцию «сессия НЗСН не видит файлы Причала» обеспечиваем **мы** (отдельные UNIX-пользователи/контейнеры на сессию, ограничение cwd, allow-list инструментов, хуки-перехватчики). Это инфраструктурная задача, не «из коробки». **[вывод R&I; источником прямо не закрыто — пометить как требование к фазе безопасности]**
- Категории «красной зоны» из §7.2 концепта (деплой, `git push --force`, `reset --hard`, rm, внешние интеграции) технически реализуются хуком `PreToolUse` + `permission_mode` + кастомным permission-каллбеком. **[подтверждено наличием механизма; конкретные сигнатуры — на этапе реализации]**

## 4. Практичная рекомендация: чем держать «мозг» 24/7 на VPS

**Подтверждаю выбор архитектора — вариант A (Claude Agent SDK как долгоживущие процессы на нашем VPS) как основной движок.** Корректировки — ниже.

Конкретика «как именно держать 24/7 и стримить наружу» (раньше была помечена в ADR/концепте как «нельзя брать по памяти» — теперь закрыто источником):
1. **Python `ClaudeSDKClient` как форма chat-core.** Это объект, который держит сессию открытой внутри процесса и продолжает диалог без ручной передачи ID — естественная опора для «движка сессии». (В TS такого session-holding объекта нет — там `continue: true` на каждый вызов; экспериментальный V2 session API с `createSession()`/`send`/`stream` **удалён** в TS SDK 0.3.142. Это аргумент в пользу **Python для chat-core**, если выбирать язык серверного движка.) **[подтверждено]**
2. **Persistence для переживания рестартов:** состояние пишется в JSONL автоматически; возобновление — `resume=<session_id>`. ID ловим из `ResultMessage.session_id` / init-`SystemMessage`. Менеджер сессий хранит карту «pod → session_id» и при рестарте поднимает `resume`. **[подтверждено]**
3. **Стрим в наш gateway:** итерируем сообщения движка, `include_partial_messages=True` для живого набора текста, хуки для событий-действий и аудита; gateway перекладывает это в WSS-кадры для тонких клиентов — ровно как описано в §5.2 концепта. **[подтверждено]**
4. **Догрузка после обрыва (§5.4 концепта):** наш слой нумерует и хранит события — это **наша** ответственность в любом случае (у SDK-стрима «переигровки по умолчанию нет», как и у SSE Managed Agents). Заложено в концепте верно. **[подтверждено принципиально]**
5. **Гибрид (вариант C) усилился.** Поскольку у Managed Agents теперь есть **self-hosted sandbox**, будущая песочница «на пробу» может жить на нашей инфраструктуре, а не только в облаке Anthropic. Это снимает часть возражения ADR против B (про «чужой контейнер и секреты»). Рекомендую в формулировке «тип среды» сессии предусмотреть три значения: `local-VPS-direct` (A, основной) / `managed-self-hosted` / `managed-cloud`. **[подтверждено наличием self-hosted; рекомендация R&I]**

**Резюме рекомендации:** A — основной движок (подтверждён и теперь технически детализирован источником). Менеджер сессий = карта pod→session + изоляция по cwd/правам + нумерованный журнал событий. Язык chat-core — **Python (`ClaudeSDKClient`)** предпочтительнее из-за наличия session-holding клиента. C (гибрид) — задел, и он стал привлекательнее благодаря self-hosted sandbox.

## 5. Корректировки к ADR-0001 / концепту (что поправить перед Фазой 0)

1. **Факт «environments только cloud» устарел.** Есть self-hosted sandbox на своей инфраструктуре. → Поправить ADR §Решение-1 (вариант B) и концепт §4.1/таблицу §4.2: B больше не означает обязательный уход кода/секретов к Anthropic.
2. **«Detali Agent SDK нельзя брать по памяти» — закрыто.** Долгоживущая сессия (`ClaudeSDKClient`), resume, стрим, суб-агенты, хуки — всё подтверждено официально (ссылки в §2). → Снять пометку «требует проверки» в концепте §4.3 и риск «Точность возможностей Agent SDK» в §11.1 можно понизить до «низкий».
3. **Уточнить, что Agent SDK ≠ песочница ОС.** Изоляция сессий проектов между собой — наша инфраструктурная задача (UNIX-пользователи/контейнеры), а не свойство SDK. → Добавить как явное требование в фазу безопасности (Фаза 6) и в §7.3.
4. **Новый эксплуатационный риск — лимит Agent SDK на подписках.** С 15.06.2026 использование Agent SDK и `claude -p` на подписочных планах тянет из **отдельного месячного Agent SDK-кредита**, отдельного от интерактивных лимитов. → Внести в §11.1 рисков (см. §7.1 ниже). **[подтверждено]**

## 6. Риски и ограничения (подтверждённые источником)

### 6.1. Стоимость / лимиты
- **[подтверждено]** С **15 июня 2026** Agent SDK и `claude -p` на подписочных планах списываются из нового **месячного Agent SDK-кредита**, отдельного от интерактивных лимитов. «Мозг 24/7» с несколькими параллельными сессиями будет жечь этот кредит постоянно — нужно заранее понять, хватит ли подписки или нужен API-ключ с pay-as-you-go. **Бизнес/бюджетное решение Владельца** (через Координатора).
- **[подтверждено]** Managed Agents имеет отдельные rate limits (страница reference) — детально не выгружал, при выборе B сверить.

### 6.2. Данные и комплаенс
- **[подтверждено]** Managed Agents **stateful by design** → **не подпадает** под Zero Data Retention и **не покрывается** HIPAA BAA. Для гибрида (B) это ограничение учесть: история сессий и состояние хранятся на стороне Anthropic (для cloud-варианта). Для A — всё у нас, ZDR-вопрос не возникает.

### 6.3. Версионирование SDK
- **[подтверждено]** SDK активно меняется: V2 session API (`createSession`/`send`/`stream`) **удалён** в TS 0.3.142. Менять фундамент под deprecated/экспериментальные API нельзя. → Зафиксировать минимальную версию SDK и читать CHANGELOG (TS и Python — ссылки в overview) перед апгрейдами. Архитектурно опираться только на стабильные `query()` / `ClaudeSDKClient` / опции сессий.

### 6.4. Аутентификация движка
- **[подтверждено]** Anthropic **не разрешает** сторонним разработчикам предлагать claude.ai-логин/лимиты в своих продуктах на Agent SDK без отдельного одобрения. Для «Мостика» (личный продукт Владельца, не публичный SaaS) — использовать API-ключ. Если когда-нибудь пойдём в multi-user-as-product — сверить условия. → Пометка для Фазы 7 (multi-user).

### 6.5. VPS — единая точка отказа
- Не источник, а архитектура: «мозг» на одном VPS. Заложено в ADR (бэкапы состояния, мониторинг, резерв). Persistence сессий (JSONL + resume) этому помогает: после восстановления VPS сессии поднимаются по `resume`. **[механизм подтверждён]**
