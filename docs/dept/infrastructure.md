- **Статус:** черновик на согласование Координатора → Владельца
- **Дата:** 2026-06-13
- **Автор:** Директор по инфраструктуре coordinata56
- **Родительские документы:** `/root/m-orbital/docs/CONCEPT_bridge_owner_cockpit.md`, `/root/m-orbital/docs/adr/0001-engine-and-thin-client.md`
- **Границы документа:** только проектирование. Ничего не разворачиваем, продакшн не трогаем, агентов не запускаем, не коммитим. Все живые действия — после явного одобрения Владельца через Координатора.

## 0. Что уже стоит на земле (bootstrap-аудит, факты, не память)

Аудит текущего общего VPS холдинга (хост `ams-1-vm-57wm`, он же `81.31.244.71`):

- **Железо:** 8 vCPU, 15 ГБ RAM, диск 154 ГБ (свободно ~67 ГБ). Средняя загрузка (load average) ~1–3 из 8 — запас по процессору есть, по диску комфортно.
- **На одном ядре уже живут (root, Docker):** coordinata56 (backend+frontend+postgres+onec-mock), НЗСН HR (backend+postgres), НЗСН ЦС (backend+postgres), FBTO (4 сервиса + Directus + postgres), плюс `audit_test_pg`. Итого ~13 контейнеров.
- **Долгоживущие процессы вне Docker:** `claude-telegram.service` (Claude Code в tmux под systemd, `Restart=always`), `coordinata56-cardinal.service`, `coordinata56-dashboard.service`.
- **Готовый паттерн «руки с подтверждением»:** `/srv/M_ROOT_EXECUTOR/` — approval-gated исполнитель root-команд с `THREAT_MODEL.md`, `SECURITY.md`, hardening в юните (`NoNewPrivileges`, `ProtectHome`, `ProtectSystem=full`, `PrivateTmp`). Это прямой архитектурный предок «красной зоны» Мостика — переиспользуем подход, а не изобретаем.
- **Бэкапы:** systemd-таймеры `*-backup.timer` для НЗСН CRM/CS/HR и FBTO; артефакты в `/root/backups/<pod>/`. Паттерн «таймер + дамп + ротация» уже работает — его и тиражируем на состояние Мостика.
- **Секреты:** `/root/.claude/secrets/*.env`, режим `0600`, вне git. База для хранилища секретов есть.

**Главный вывод аудита:** холдинг де-факто уже работает по модели «движок ИИ + руки на VPS» — `claude-telegram.service` это ровно она, только без UI и без изоляции по проектам. Мостик не вводит новый класс системы, он **дисциплинирует и одевает в лицо** уже существующую практику. Это снижает технический риск Фазы 0.

## 1. Где размещать «мозг»: общий VPS vs выделенный сервер (РАЗВИЛКА ВЛАДЕЛЬЦА)

### Формулировка развилки
«Мозг» Мостика — это сессии ИИ с правами на правку файлов, shell и деплой **всех 8 pod'ов холдинга**. Вопрос: ставить его на текущий общий VPS (где уже крутится боевой HR/ЦС/coordinata56/FBTO) или на отдельный выделенный сервер?

Это не чисто техническое решение — это про **степень изоляции самого ценного и самого опасного компонента холдинга** от боевой нагрузки. Цена ошибки/взлома «мозга» максимальна: он по определению дотягивается до всего.

### Сравнение

| Критерий | Общий VPS (текущий) | Выделенный сервер под «мозг» |
|---|---|---|
| Доступ к боевым папкам `/root/<pod>/` | Прямой, локальный (родная модель A из ADR) | Через сеть/монтирование — сложнее, медленнее, новый класс рисков |
| Изоляция критичной инфраструктуры | **Низкая:** взлом «мозга» = тот же хост, что боевой HR/ЦС | **Высокая:** компрометация мозга не открывает напрямую боевые БД |
| «Радиус поражения» при ошибке ИИ | Весь хост (включая боевые контейнеры HR/ЦС) | Ограничен сервером мозга + тем, до чего он дотянется по сети |
| Бюджет | 0 ₽ доп. (используем имеющиеся ресурсы) | +стоимость VPS (ориентировочно тариф уровня текущего) |
| Ресурсный конфликт | Сессии ИИ прожорливы; конкуренция за CPU/RAM с боевыми сервисами | Нет конкуренции с боевыми |
| Сложность старта (Фаза 0) | Минимальная — всё уже здесь | Выше — provisioning, сеть, доступ к pod'ам |
| Единая точка отказа | Падёт хост — встанут и боевые сервисы, и мозг | Падёт мозг — боевые сервисы живут дальше |

### Рекомендация Директора (двухступенчатая)

**MVP (Фазы 0–2): общий VPS, но с жёсткой изоляцией процессов (см. §2).** Обоснование:
1. Модель A из ADR требует **локального** доступа к `/root/<pod>/`. На отдельном сервере боевые папки пришлось бы монтировать по сети — это и сложнее, и сам по себе новый вектор атаки (сетевой доступ к чужим боевым данным). Для MVP это противоречит принципу минимальной достаточности.
2. Холдинг уже так живёт (`claude-telegram.service`). Старт «на земле» — нулевой риск нового железа и нулевой бюджет.
3. Изоляцию даём не сервером, а уровнем процесса (отдельный системный пользователь + namespaces + лимиты, §2). Это закрывает 80% угрозы «сессия одного проекта дотянулась до другого» без затрат.

**Полный продукт (Фаза 6, «закалка»): выделенный сервер под мозг — рекомендуется.** Когда Мостик станет постоянным флагманом с потенциальным multi-user, концентрация «рук на всё» на одном хосте с боевым HR/ЦС — неприемлемый системный риск. К этому моменту:
- мозг переезжает на отдельный VPS;
- к боевым pod'ам он ходит контролируемо (отдельный сетевой контур, а не общий localhost);
- падение/компрометация мозга больше не роняет боевые сервисы холдинга.

**Что нужно от Владельца сейчас:** подтвердить курс «MVP на общем VPS с процессной изоляцией → выделенный сервер на фазе закалки». Если Владелец хочет выделенный сервер сразу с MVP (приоритет изоляции над скоростью/бюджетом) — это его право, тогда мы закладываем сетевой доступ к pod'ам с первой фазы (дороже и дольше, но чище по безопасности).

## 2. Изоляция сессий проектов друг от друга

Цель: сессия НЗСН **физически** не может прочитать файлы Причала и запустить его агентов. Это и защита (взлом одной сессии не открывает остальные), и соблюдение уже принятого правила «кросс-проектные вызовы запрещены».

Три уровня изоляции, от обязательного к усиленному:

### Уровень 1 — отдельный системный пользователь на сессию (обязательно с MVP)
- Каждая сессия проекта запускается **не под root**, а под выделенным служебным пользователем (`bridge-coordinata56`, `bridge-nzsn-cs`, …).
- Папка проекта `/root/<pod>/` принадлежит этому пользователю (или доступна только ему по группе); чужие папки — закрыты правами файловой системы (Unix-права/ACL).
- Это базовый, дешёвый, надёжный барьер: даже без контейнеров взлом сессии НЗСН упирается в права доступа к файлам Причала.
- **Важное последствие:** сегодня всё крутится под root — это придётся менять. Понижение привилегий сессий — отдельный пакет работ (см. §6, devops).

### Уровень 2 — Linux namespaces / контейнеризация рабочей среды (рекомендуется)
- Каждая сессия — в своём контейнере (или `systemd-nspawn`/контейнер с bind-mount только своей папки pod'а). «Namespace» (термин) = изолированное представление системы: процесс видит только свою файловую систему, своих соседей по процессам, свою сеть.
- Внутрь монтируется **только** `/root/<pod>/` этого проекта + его конфиг агентов. Остальной диск для сессии не существует.
- Сеть сессии — ограничена: наружу только то, что нужно (API Anthropic, git-remote pod'а), не весь localhost с чужими БД.

### Уровень 3 — изоляция набора инструментов и агентов (на уровне менеджера сессий)
- Сессия получает только агентов с префиксом своего pod'а (правило холдинга уже это диктует).
- «Красная зона» действий (деплой, разрушительный git, удаление, внешние интеграции) проходит через единый approval-шлюз — переиспользуем модель `M_ROOT_EXECUTOR`.

**Рекомендация:** MVP — Уровень 1 (отдельные пользователи) как минимум обязательный; Уровень 2 (namespaces/контейнеры) — целевой к Фазе 3 (несколько сессий). Уровень 3 — с первой фазы, потому что это и есть «руки с подтверждением».

> Изоляция Координатора холдинга — особый случай: ему нужен межпроектный **обзор** (read-only сводки), но не руки в чужих папках. Технически: read-доступ к реестру/статусам, без write в `/root/<pod>/`. Раздаёт задачи проектным сессиям, не правит сам.

## 3. Процесс-менеджмент 24/7

Требование концепта: сессии живут на VPS постоянно, переживают перезапуск сервера, продолжают работу при выключенных устройствах Владельца.

### Supervisor: systemd (переиспользуем готовый паттерн)
- Эталон уже есть — `claude-telegram.service`: systemd-юнит, `Type=forking`, процесс в `tmux`, `Restart=always`, `RestartSec=30`. Это ровно «процесс пережил падение и сам поднялся».
- **Менеджер сессий** Мостика — это надстройка над тем же подходом: один управляющий systemd-сервис (`bridge-session-manager.service`), который ведёт реестр сессий и поднимает/гасит их. Каждая сессия проекта — отдельный управляемый процесс (под своим пользователем, §2), который менеджер умеет разбудить/погасить.
- При перезагрузке VPS: `WantedBy=multi-user.target` поднимает менеджер сессий → он по своему реестру восстанавливает сессии в последнем сохранённом состоянии (см. §4 — состояние на диске/в БД, не в памяти процесса).

### Переживание перезапуска (resume)
- Ключевой нюанс: процесс ИИ умирает при перезагрузке. «Пережить» = **восстановить состояние из персистентного хранилища**, а не «не умирать». Поэтому состояние сессии (история разговора, очередь действий, ожидающие подтверждения) пишется на диск/в БД непрерывно, а не держится только в RAM.
- Возобновление сессии Agent SDK (resume) — точный механизм **сверяем с актуальной офиц. документацией Agent SDK перед реализацией Фазы 0** (помечено в ADR §4.3; по памяти не берём).

### Логи
- Каждый сервис → `journald` (как сейчас: `StandardOutput=journal`). Плюс отдельный структурированный журнал событий сессии (для догрузки клиентами после обрыва, §5 концепта) и журнал аудита действий (§5 ниже).
- Ротация логов — стандартная (`journald` + logrotate для файловых журналов), чтобы диск не заполнялся.

### Ресурсные лимиты (критично — мозг прожорлив, рядом боевой HR/ЦС)
- На каждую сессию — лимиты через systemd (`MemoryMax`, `CPUQuota`, `TasksMax`) или cgroups. Цель: одна разогнавшаяся сессия не съедает RAM/CPU у боевых сервисов на общем VPS.
- Глобальный потолок для всего стека Мостика, чтобы он не конкурировал с HR/ЦС за ресурсы. Это прямое следствие выбора «общий VPS» в §1.

## 4. Деплой продукта, обновления, бэкапы состояния

### Деплой компонентов
- **gateway + session-manager + chat-core** — серверные, на VPS. Деплой: git-pull в `/root/m-orbital` (или отдельный `/srv/bridge/`) + рестарт systemd-юнитов. Контейнеризация серверной части — желательна (повторяемость), но для Фазы 0 допустим прямой запуск под systemd, как `claude-telegram`.
- **Клиенты:** Mac (Tauri) — сборка артефакта + авто-обновление приложения; mobile PWA — статика на Cloudflare/раздача с VPS. Сборка клиентов отдельна от серверного деплоя.
- **Принцип:** обновление серверного «мозга» не должно ронять активные сессии без предупреждения. Заложить graceful-рестарт (дождаться сохранения состояния → рестарт → восстановление).

### Бэкапы (переиспользуем паттерн `*-backup.timer`)
Что бэкапим (это новый, ценный класс данных — личная история работы Владельца с ИИ):
1. **История чатов и состояние сессий** — переписка, решения, очередь действий, ожидающие подтверждения. Самое ценное и невосстановимое.
2. **Журнал аудита действий** — что руки ИИ делали (для разбора инцидентов; см. §5).
3. **Реестр сессий и их конфигурация.**

Как: systemd-таймер `bridge-state-backup.timer` (по образцу `nzsn-hr-backup`), дамп в `/root/backups/bridge/` с ротацией, плюс выгрузка в офсайт (S3/внешнее хранилище) — потому что история работы Владельца не должна жить только на единой точке отказа (см. риск VPS).

### Хранилище состояния
- Состояние сессий — в постоянном хранилище (БД или файлы с журналом событий). Конкретный выбор (PostgreSQL отдельной БД vs встраиваемое хранилище) — **пакет для db-head** (§6): схема истории чатов, журнала событий с порядковыми номерами, очереди подтверждений, аудита.

## 5. Безопасность периметра

Инструмент с руками — цена ошибки/взлома максимальна. Безопасность с первой строки.

### Доступ только Владельца
- **Аутентификация:** биометрия на устройстве (Face ID/Touch ID) + серверный токен с коротким сроком жизни и обновлением. Секрет — не в коде, не в git (правило холдинга).
- **Привязка устройства:** каждое устройство регистрируется явно; новый клиент не пускается к сессиям без подтверждения с уже доверенного устройства.
- v1 — один пользователь; модель прав закладываем multi-user-ready (Пользователь→Роль→Право), но без преждевременной реализации.

### Защита канала
- Только **WSS** (WebSocket поверх TLS — шифрованное двустороннее соединение). Никакого открытого WS/HTTP к gateway.
- TLS-терминация на reverse-proxy (nginx/Caddy) перед gateway; сертификат Let's Encrypt. Gateway наружу — только через прокси, сам порт gateway слушает localhost.
- Все боевые сервисы холдинга уже привязаны к `127.0.0.1` (видно в аудите портов) — тот же принцип для gateway: наружу торчит только прокси.

### Хранение секретов
- По образцу `/root/.claude/secrets/*.env` (режим `0600`, владелец root/служебный пользователь, вне git).
- Секреты сессии доступны только её пользователю (§2, Уровень 1) — сессия НЗСН не читает секреты Причала.
- На фазе закалки — отдельное хранилище (vault-подход), но для MVP файлов с жёсткими правами достаточно (skeleton-first, как по всему холдингу).

### Аудит действий «рук»
- Каждое действие ИИ (правка файла, shell, git, деплой, запуск агента) → в **журнал аудита**: что, когда, в каком проекте, по чьей команде, чем закончилось. ПД в логах маскируется всегда.
- Переиспользуем модель `M_ROOT_EXECUTOR` (`THREAT_MODEL.md`/`SECURITY.md` уже написаны) для «красной зоны»: опасное действие → запрос подтверждения Владельцу → исполнение только после явного «да» → запись в журнал executed/.
- «Красная зона» по умолчанию максимально осторожная: деплой, `git push --force`/`reset --hard`, удаление, правка системных файлов, любые живые внешние интеграции (банки/Росреестр/1С-push). Граница — конфигурируема, **развилка Владельца** (§11.2.3 концепта).

## 6. Декомпозиция на пакеты и вопросы Владельцу

### Пакеты для `devops-head` → `devops`
- **DEVOPS-1 (Фаза 0):** прототип `bridge-session-manager.service` по образцу `claude-telegram.service` — одна сессия (coordinata56) под systemd, `Restart=always`, journald-логи. Доказать «процесс живёт 24/7 и поднимается после рестарта». *Только проектирование/прототип на dev, не на боевых данных.*
- **DEVOPS-2:** понижение привилегий — служебные пользователи `bridge-<pod>`, права на папки, отбор от root. Самый чувствительный пакет (трогает текущую root-модель) — отдельно, с планом отката.
- **DEVOPS-3:** ресурсные лимиты (cgroups/systemd `MemoryMax`/`CPUQuota`) на сессии + глобальный потолок стека Мостика, чтобы не конкурировал с боевыми HR/ЦС.
- **DEVOPS-4:** контейнеризация сессий (namespaces/контейнер с bind-mount только своей папки) — целевой к Фазе 3.
- **DEVOPS-5:** reverse-proxy + TLS (WSS) перед gateway; gateway на localhost; ревизия firewall.
- **DEVOPS-6:** бэкапы состояния — `bridge-state-backup.timer` + офсайт-выгрузка, по образцу `nzsn-hr-backup`.

### Пакеты для `db-head` → `db-engineer`
- **DB-1:** схема хранения состояния сессий — история чатов, журнал событий с **порядковыми номерами** (для догрузки после обрыва, §5 концепта), очередь подтверждений (переживающая обрыв), реестр сессий.
- **DB-2:** схема журнала аудита действий рук (что/когда/проект/команда/результат) + политика маскирования ПД.
- **DB-3:** решение «отдельная БД vs встраиваемое хранилище» для состояния Мостика; индексы под выборку истории и догрузку по номеру события.
- *Все миграции — round-trip через Директора инфраструктуры перед применением (правило холдинга).*

### Связка с `integrator-head`
- На MVP внешних интеграций у Мостика нет (движок A — локальный). Точка касания — позже, если мозг ходит к внешним API; тогда — через approval-шлюз и без живых вызовов до production-gate (правило «no live external integrations»).

### Вопросы Владельцу (через Координатора)
1. **Размещение VPS (§1):** подтвердить курс «MVP на общем VPS с процессной изоляцией → выделенный сервер на фазе закалки»? Или выделенный сервер сразу (приоритет изоляции над скоростью/бюджетом)?
2. **Понижение привилегий (§2/DEVOPS-2):** согласны, что сессии ИИ перестают работать под root и переходят на служебных пользователей? Это повышает безопасность, но требует аккуратной миграции текущей модели.
3. **Граница «красной зоны» (§5):** подтвердить осторожный список по умолчанию (деплой, разрушительный git, удаление, системные файлы, внешние интеграции — всегда спрашивать)?
4. **Офсайт-бэкап истории (§4):** куда выгружать копию личной истории работы с ИИ (S3-совместимое внешнее хранилище)? Это чувствительные личные данные — нужно решение Владельца о месте хранения.
