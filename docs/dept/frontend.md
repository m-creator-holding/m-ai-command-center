**Опирается на:**

> Документ только проектный: архитектура клиентов и декомпозиция работ. Кода нет, агенты не запускаются, ничего не коммитится. Принцип — сначала рабочий минимум (MVP), потом полировка. ## 0. Что важно держать в голове

«Мостик» для фронтенда — это **тонкий клиент**: он почти ничего не вычисляет сам, а показывает то, что прислал сервер (Шлюз), и отправляет обратно команды Владельца. Значит, основная сложность фронтенда — не «бизнес-логика», а **надёжное реальное время**: поток событий от ИИ, переподключение без потери истории, синхронизация двух устройств, переживающие обрыв подтверждения.

Второй стержень — **максимум общего TypeScript-кода** между Mac (Tauri) и телефоном (PWA→React Native). Платформенным остаётся только «корпус».

## 1. Архитектура клиентов: общее ядро + две оболочки

### 1.1. Три уровня кода

```
┌─────────────────────────────────────────────────────────────┐
│  ОБЩЕЕ ЯДРО (TypeScript, не зависит от платформы)            │
│  • protocol-client — WSS-клиент: connect/reconnect,          │
│       журнал событий с номерами, догрузка пропущенного,      │
│       очередь подтверждений                                  │
│  • state — модель данных и хранилище состояния (стор):       │
│       проекты, сессии, разговоры, очередь действий           │
│  • api — типобезопасные обёртки над кадрами Шлюза            │
│  • ui-kit — переиспользуемые React-компоненты слоёв 2 и 3     │
│       (рабочий стол, чат, лента событий) + контракт темы     │
└─────────────────────────────────────────────────────────────┘
        ▲                                          ▲
        │                                          │
┌───────┴──────────────┐              ┌────────────┴───────────┐
│ ОБОЛОЧКА Mac (Tauri)  │              │ ОБОЛОЧКА mobile         │
│ • окно, иконка в доке │              │ • MVP: PWA (тот же web) │
│ • биометрия (Touch ID)│              │ • потом: React Native   │
│ • хостит Орбиту       │              │ • пуши, Face ID         │
│   (Three.js, как есть)│              │ • адаптив: вкладки/жесты│
└───────────────────────┘              └─────────────────────────┘
```

**Принцип границы:** всё, что про «что показать и какие данные», — в общем ядре. Всё, что про «как устройство устроено» (окно, пуш, биометрия, файловая система), — в оболочке. Одна правка в логике чата автоматически попадает на оба устройства.

### 1.2. Что общее, что специфичное

| Слой | Общий код (TS) | Специфика Mac (Tauri) | Специфика mobile |
|---|---|---|---|
| Протокол/состояние | весь (protocol-client, state, api) | — | — |
| Чат (Этаж 3) | весь UI чата и ленты действий | — | компоновка: лента действий снизу, не сбоку |
| Рабочий стол (Этаж 2) | весь UI (карточки, TraceTree-обёртка) | — | вкладки вместо колонок |
| Орбита (Этаж 1) | модель данных pod'ов + переходы | хостит Three.js-сцену как есть | на телефоне — упрощённый список/2D (см. §2.4) |
| Вход/безопасность | хранение токена (в памяти/secure storage) | Touch ID, привязка устройства | Face ID, пуши |
| Корпус | — | окно, док, авто-обновление | экран, статус-бар, жесты |

> **Стек, который рекомендую зафиксировать:** TypeScript + React + Vite (Орбита уже на Vite, переиспользуем тулинг), стор — лёгкий (Zustand) без тяжёлого Redux (принцип «сначала простота»). Tauri для Mac. PWA на том же web-бандле для MVP-мобильного. Точные версии Tauri/Agent SDK сверяются с актуальной документацией на старте реализации (помечено в ADR §4.3).

## 2. Карта экранов трёх слоёв и встраивание активов

Логическая структура у всех трёх визуальных вариантов **одна и та же** (см. мокапы `/root/command-center-mockups/`): обзор проектов → рабочая область проекта → чат с Помощником. Различается только подача. Это и доказывает, что слой 2 и 3 строятся один раз, а вариант A/B/C — это тема (§4).

### 2.1. Этаж 1 — Орбита (обзор холдинга)

| Элемент | Источник | Что делает фронтенд |
|---|---|---|
| 3D-сцена сфер | `m-orbital/src/main.js` (Three.js, bloom, камера) — **оставляем как есть** | оборачиваем в React-компонент `<OrbitalStage>` |
| Узлы = 8 pod'ов | сейчас захардкожен массив `DEPARTMENTS` | заменяем источник: данные о pod'ах приходят живьём от Шлюза (WSS), не из константы |
| Здоровье (цвет/пульс) | функции `statusHex/statusCSS` уже есть | мапим статус pod'а (тревога/идут агенты/ждёт решения) на существующие состояния |
| Клик по сфере | OrbitControls + raycast | эмитим событие `openProject(podId)` → переход на Этаж 2 |
| Слой «Координатор холдинга» | новый | отдельная «центральная» сфера/панель — сводка по всему холдингу |

**Минимальные правки Орбиты (не переписывание):** (1) вынести `DEPARTMENTS` из константы в проп/подписку на стор; (2) узлы — pod'ы, а не департаменты; (3) проброс клика наружу. Рендер, постобработка, управление камерой — **не трогаем**.

### 2.2. Этаж 2 — Рабочий стол проекта

Экраны/блоки (видны во всех трёх мокапах):
- шапка проекта (имя, отрасль, число агентов в работе);
- KPI-полоса (открытые задачи, лиды, страницы, последний деплой) — заполняется данными сессии;
- вкладки **Обзор / Задачи / Активность / Команда агентов**;
- блок «Задачи» с статусами (блокер/в работе/готово);
- блок «Последняя активность» (лента событий проекта);
- **TraceTree** — дерево агентов проекта, переиспользуем из Кардинала (`sitroom-tracetree.js`, источник `GET /api/agent-tree`);
- список открытых вопросов/развилок к Владельцу;
- кнопка «Открыть разговор» → Этаж 3.

Встраивание Кардинала: TraceTree и голос подключаются через уже существующий **SITROOM_CONTRACT** (идемпотентные init-функции, mount-points, CSS-переменные). Мы оборачиваем их в React-компоненты-адаптеры, не переписываем внутренности.

### 2.3. Этаж 3 — Чат с ИИ (ядро MVP)

Главный экран продукта. Состоит из двух зон:
- **Переписка** (как мессенджер): сообщения Владельца и ответы ИИ, поле ввода, прерывание (стоп);
- **Живой поток действий** (отличие от чат-бота): лента «читаю файл X», «запускаю агента backend», «делаю правку», бейджи статуса инструмента (см. §3);
- **Кнопки подтверждения** опасных действий («Подтвердить / Отклонить»), переживающие обрыв связи;
- **Голос «Кардинал»** — голосовой ввод команды (запись→стоп→отправить), переиспользуем механику из дашборда; имя «Кардинал» остаётся за CEO-уровнем.

На Mac поток действий — справа/сбоку; на телефоне — снизу или отдельной вкладкой (общий компонент, разная компоновка).

### 2.4. Орбита на телефоне

3D-сцена тяжела для телефона и для жестов навигации. На MVP-мобильном Этаж 1 — **упрощённый список/2D-карта pod'ов** (как левый рейл в мокапах B/C), а не полноценный Three.js. Это не дубль логики: список и 3D читают **один и тот же стор** pod'ов. 3D на телефоне — задел на потом (RN), не в MVP.

## 3. Real-time UX: поток событий, индикаторы, обрыв, синхронизация

Это самая ответственная часть фронтенда. Закладываем с первой строки (ADR §5.4).

### 3.1. Поток событий от агентов
Шлюз шлёт по WSS типизированные **кадры**: `message` (текст ИИ/Владельца), `action` (событие-действие инструмента), `confirm-request` (запрос подтверждения), `state` (снимок состояния сессии). Фронтенд раскладывает их в две ленты: переписка и поток действий. Формат кадра не зависит от внутреннего движка — Шлюз его уже нормализовал (контракт с бэкендом, §6).

### 3.2. Индикаторы «инструмент работает»
Каждое `action`-событие имеет жизненный цикл: `started → running → done/failed`. Фронтенд показывает:
- бейдж с именем инструмента/агента и спиннером, пока `running`;
- галочку/крест по завершении;
- длительность (механика измерения уже есть в Кардинале — `tool_durations.jsonl`).
Несколько действий могут идти параллельно — лента это поддерживает.

### 3.3. Обрыв и реконнект (критично)
- **Журнал с номерами:** каждое событие сессии имеет порядковый номер; клиент помнит последний полученный.
- **Догрузка после реконнекта:** при восстановлении клиент сообщает «видел до №N» → Шлюз досылает пропущенное. Без этого кусок истории теряется (у потоков нет переигровки по умолчанию).
- **Состояние соединения** видно Владельцу: «на связи / переподключаюсь / оффлайн».
- **Оффлайн-буфер ввода:** команда, набранная без связи, ставится в очередь и уходит при реконнекте (с явной пометкой «не отправлено»).

### 3.4. Подтверждения переживают обрыв
Если ИИ ждёт «да/нет» на деплой, а связь упала — запрос не теряется и не «зависает молча»: при переподключении (на любом устройстве) он снова показывается из очереди сессии.

### 3.5. Синхронизация между устройствами
Состояние живёт на сервере; Mac и телефон подписаны на одну серверную сессию. Фронтенд:
- при подключении получает снимок состояния (`state`) и догружает события;
- два устройства онлайн → видят одно и то же синхронно (Шлюз рассылает всем окнам сессии);
- ввод одного устройства отражается на другом (например, «Владелец печатает» / отправленная команда видна сразу на обоих).

## 4. Темы и визуальные варианты (A/B/C подключаются без переписывания)

Ключевое требование Владельца: направление ещё не выбрано — нельзя завязывать логику на конкретную картинку.

### 4.1. Разделение «логика ↔ внешний вид»
Из анализа трёх мокапов: все варианты имеют одинаковые **логические зоны** (навигация проектов / рабочая область / чат) и одинаковые **данные**. Различия — чисто визуальные:
- **A «Орбита»** — кинематографичный 3D-фон, свечение, тёмный космос;
- **B «Ситуационная комната»** — плотная сетка карточек, KPI-полоса, моноширинный «командный» стиль (это фактически язык SITROOM_CONTRACT, уже существующий);
- **C «Спокойный продукт»** — чистый SaaS: левый рейл, светло-тёмные карточки, минимум эффектов.

### 4.2. Механика темы
- **Контракт темы** — единый набор токенов (цвета, типографика, радиусы, тени, плотность, наличие/сила эффектов свечения). Уже есть прецедент: CSS-переменные `--sit-*` в SITROOM_CONTRACT — берём этот же подход и расширяем.
- Компоненты слоёв 2 и 3 **не знают** о конкретной теме — читают токены. Смена темы = смена набора токенов + опциональных «эффект-слоёв» (например, 3D-фон или bloom — это флаг темы, а не часть компонента).
- **Орбита (3D)** — отдельный визуальный слой, который тема A включает как фон/основу, а темы B/C могут заменить упрощённым 2D-обзором. Логика pod'ов и переходов от этого не меняется (§2.4).
- **Плотность** (densely B vs airy C) — параметр темы (отступы, размер шрифта), а не разная верстка.

### 4.3. Следствие для работ
Слои 2 и 3 строим **один раз** на нейтральных токенах. Когда Владелец выберет направление — это пакет «тема X» (набор токенов + эффект-слои), не переделка экранов. На MVP допустимо начать с самой простой темы (C «Спокойный продукт») как базовой, чтобы не блокироваться на выборе, и подключить A/B позже.

### 6.2. Вопросы Владельцу (через Координатора)

1. **Базовая тема MVP.** Чтобы не блокировать разработку слоёв 2–3 на выборе A/B/C, предлагаю стартовать на нейтральной теме C «Спокойный продукт», а A и B подключить пакетами тем позже. Согласны начать с C как технической базы (это не финальный выбор направления)?
2. **Глубина мобильного на MVP.** Концепт допускает «PWA достаточно для MVP, мобильное может ждать». Делаем мобильное в первой волне наравне с Mac, или MVP = только Mac (Tauri) + браузерный чат, а мобильное в следующей фазе? От ответа зависит, входят ли FE-13/FE-14 в первую волну.

### 6.3. Открытый технический пункт (не к Владельцу, к фазе реализации)
Перед началом FE-2 — сверить с **актуальной официальной документацией Agent SDK** механизм выдачи потока событий наружу и resume сессии (помечено в ADR §4.3). Фронтенд завязан на формат, который нормализует Шлюз, но контракт кадров должен учитывать реальные возможности движка.

## Резюме для Координатора

- **Структура клиентов:** общее TS-ядро (protocol-client + state + api + ui-kit) и две тонкие оболочки — Mac (Tauri, иконка в доке, биометрия) и mobile (MVP — PWA на том же web-бандле, позже — React Native). Платформенным остаётся только «корпус».
- **MVP (Фазы 0–2):** рабочий чат с руками (Этаж 3) + надёжное реальное время (WSS, журнал с номерами, догрузка, переживающие обрыв подтверждения) + Mac-приложение. Пакеты FE-1…FE-8, первым — узкий vertical slice на одном проекте.
- **Орбиту встраиваем, не переписываем:** `m-orbital` оборачиваем в React-компонент, меняем только источник данных (массив `DEPARTMENTS` → живая подписка на pod'ы от Шлюза) и проброс клика. Рендер/bloom/камера — без изменений. Кардинал (TraceTree, голос) — через существующий SITROOM_CONTRACT.
- **Темы:** все три мокапа имеют одинаковые логические зоны — различие чисто визуальное. Логику строим один раз на токенах темы (`--sit-*`-подход уже есть); A/B/C подключаются как пакеты токенов + эффект-слои, без переделки экранов.
- **Топ-2 вопроса Владельцу:** (1) стартовать ли разработку на базовой теме C, не дожидаясь финального выбора направления; (2) мобильное (PWA) — в первой волне с Mac или следующей фазой.
- **Главная блокирующая зависимость:** контракт кадров WSS от бэкенда (gateway) — без него FE-1/FE-2 не стартуют.
