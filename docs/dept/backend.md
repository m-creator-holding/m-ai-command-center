**Статус:** проект на согласование (директор по бэкенду)
**Дата:** 2026-06-13
**Автор:** директор по бэкенду (L2)
**Опорные документы:**
- Концепт: `/root/m-orbital/docs/CONCEPT_bridge_owner_cockpit.md`
- ADR-0001: `/root/m-orbital/docs/adr/0001-engine-and-thin-client.md`

> Это инженерный брифинг для `backend-head` → `backend-dev`. Здесь нет кода — только проектирование модулей, контракты между ними и декомпозиция на рабочие пакеты. Детальные схемы данных и OpenAPI/JSON-схемы рисуются на фазе реализации совместно с db-engineer. Всё, что нельзя утверждать по памяти про Claude Agent SDK, помечено **«требует проверки по офиц. документации»**.
>
> Принцип над всем: **сначала узкий рабочий вертикальный срез (Фаза 0), потом наращивание.** Не строим все 8 сессий и Орбиту сразу. Против over-engineering.

## 0. Что уже зафиксировано (не пересматриваем на уровне бэкенда)

Из концепта и ADR-0001 — это вход, а не предмет обсуждения:

1. **Движок ИИ — вариант A:** Claude Agent SDK / Claude Code как долгоживущие процессы на нашем VPS. Гибрид (Managed Agents) — заложенная опция через параметр сессии «тип среды», не в MVP.
2. **Модель — «мозг на сервере + тонкие клиенты»:** вся логика и инструменты на VPS, клиенты только показывают и шлют команды.
3. **Real-time канал — WSS основной, SSE запасной** (односторонняя выдача прогресса).
4. **Сессии — трёхуровневое дерево** (уточнение Владельца msg 6844, см. §3): Координатор холдинга → сессия пода (направление) → сессия проекта. У каждого узла своя история/контекст/права; изоляция по уровням.
5. **v1 — один пользователь (Владелец), multi-user-ready** на уровне модели данных.
6. **MVP = Фазы 0–2** (рабочий чат с руками + Mac-приложение в доке).

Зона ответственности бэкенда — **нижняя половина схемы концепта (§3.4):** Шлюз, Менеджер сессий, обёртка движка (chat-core), слой Координатора, хранилище истории, событийная шина. UI-модули (orbital-ui, project-workspace, chat-ui, mac-shell, mobile-client) — департамент фронта; бэкенд отдаёт им контракт.

## 1. Серверные модули и ответственность

Семь модулей концепта, из них **шесть серверных — наши**. Ниже — детализация ответственности именно бэкенда.

### 1.1. Шлюз (`gateway`)
Единственная дверь снаружи. Всё, что приходит с устройств, и всё, что уходит на устройства, проходит через него.
- **Аутентификация и сессия устройства:** проверка токена, привязка устройства, отказ незнакомым клиентам (детали — §4 безопасности и департамент инфра).
- **Терминирование real-time:** держит WSS-соединения от всех окон (Mac + телефон + веб). SSE — запасной endpoint для «только смотреть».
- **Мультиплексирование:** одно WSS-соединение от клиента может быть подписано на несколько сессий проектов одновременно (Владелец видит орбиту + открытый чат). Шлюз маршрутизирует кадры по `session_id`.
- **Fan-out (веер рассылки):** событие одной сессии рассылается на ВСЕ онлайн-окна, подписанные на эту сессию (синхрон Mac↔телефон).
- **Очередь подтверждений:** держит «красную зону» — запросы ИИ на опасные действия, ждущие «да/нет» Владельца; переживают обрыв связи (§3.5).
- **Журнал аудита:** каждое действие ИИ + каждая команда Владельца + каждое подтверждение пишутся в аудит (§4.4).
- **Чего НЕ делает:** не думает, не исполняет инструменты, не знает внутренний формат движка. Только перекладывает и охраняет.

### 1.2. Менеджер сессий (`session-manager`)
Диспетчер «ИИ-сотрудников». Знает все сессии, их состояние и жизненный цикл.
- **Реестр сессий — ИЕРАРХИЧЕСКИЙ (дерево, §3):** Координатор холдинга → сессии подов (направлений) → сессии проектов внутри пода. Для каждого узла — `session_id`, уровень, путь в дереве (`pod_id`/`project_id`), область видимости (набор путей), статус (§3.1), параметр «тип среды». У каждого узла своя персистентная история и контекст.
- **Жизненный цикл:** создать / разбудить (resume) / приостановить / погасить процесс движка (§3) — единообразно для узла любого уровня; узлы создаются лениво.
- **Изоляция по уровням (§3.2):** при создании сессии задаёт область видимости — координатор: реестр подов (read); под: пути всех своих проектов (может быть несколько репозиториев); проект: одну свою папку. Плюс набор разрешённых агентов (префикс пода) и набор прав. Сессия физически не выходит за свою область (§4.3).
- **Состав дерева — config-as-data (§3.6):** поды и проекты описаны данными; новый под (ЖК Дарвин) или проект — правка конфигурации, не релиз.
- **Владелец персистентности:** держит состояние сессии и журнал событий с порядковыми номерами; обеспечивает восстановление после обрыва клиента и докачку (§3).
- **Чего НЕ делает:** не общается с клиентами напрямую (только через Шлюз), не лезет в файлы проектов руками.

### 1.3. Обёртка движка ИИ (`chat-core`, engine adapter)
Тонкая прослойка между нашим миром (Менеджер сессий) и Claude Agent SDK / Claude Code. **Самый чувствительный модуль** — изолирует нас от деталей и версий SDK.
- **Запуск/возобновление процесса движка** для конкретной сессии с привязкой к её папке. **Требует проверки по офиц. документации:** точный способ запуска Agent SDK как долгоживущего серверного процесса и механизм `resume` (возобновление прерванной сессии по идентификатору).
- **Канонизация событий:** движок отдаёт свой внутренний поток (мысли, вызовы инструментов, результаты). Адаптер переводит его в **наш стабильный формат событий** (§2.3), чтобы при смене версии SDK не менялся контракт наружу. Это ключевая причина существования адаптера.
- **Перехват «красной зоны»:** опасный вызов инструмента (деплой, разрушительный git, удаление, внешняя интеграция) адаптер НЕ пропускает в исполнение, а превращает в `confirmation.request` и ставит на паузу до ответа Владельца (§4.2). **Требует проверки:** механизм Agent SDK для перехвата/одобрения вызова инструмента до его исполнения (hook / permission-callback).
- **Прокидывание команд и подтверждений** от Владельца внутрь движка.
- **Реализация:** вариант A (Agent SDK на VPS). «Тип среды» — параметр; managed-Anthropic не в MVP.

### 1.4. Слой Координатора холдинга (`coordinator-session`) — корень дерева
Особая сессия (уровень 1, §3.0), технически — та же `chat-core`, но с расширенным контекстом.
- Запускается из `/root/` (как Hub-сессия), видит реестр всех подов.
- **Не операционка:** межпроектные/стратегические запросы, сводный обзор для Орбиты (Этаж 1), маршрутизация задач вниз по дереву (в сессию пода или проекта).
- **Не правит чужое без явного делегирования** — раздаёт задачи, как Президент в действующем регламенте.
- На уровне бэкенда это **корневой узел реестра-дерева** с особым профилем прав (read реестра подов, право ставить задачу нижестоящему узлу — НЕ право править его файлы).

### 1.5. Хранилище истории чатов (`history-store`)
Память каждого УЗЛА ДЕРЕВА живёт отдельно — и у пода, и у каждого его проекта, и у координатора.
- **Персистентность на узел:** сообщения, события действий, принятые решения/развилки, привязанные к `session_id` (+ путь `pod_id`/`project_id`).
- **Источник для Этажа 2** (история прошлых разговоров — отдельно по поду в целом и по каждому проекту) и для докачки событий после обрыва.
- **Изоляция по дереву:** история проекта «Сайт» недоступна проекту «HR-отдел» того же пода; история пода НЗСН недоступна поду Причал.
- **MVP-реализация — простая:** добавляемый журнал (append-only) на диске VPS (например, JSONL по сессии — тот же подход, что `agent_tree.jsonl` у «Кардинала»). СУБД и индексы — только когда упрёмся в объём/поиск, не раньше. **Решение о хранилище согласуется с db-engineer на фазе реализации.**

### 1.6. Событийная шина для real-time (`event-bus`)
Внутренний транспорт событий от сессий к Шлюзу.
- **Один поток на сессию** с монотонными порядковыми номерами (`seq`) — основа докачки.
- **MVP — внутрипроцессно / лёгкий брокер:** не тащим Kafka/RabbitMQ. Достаточно внутренней очереди процесса или лёгкого pub/sub. Тяжёлый брокер — только при реальной многосессийной нагрузке (Фаза 6). Против over-engineering.
- Шлюз — потребитель шины, клиенты — потребители Шлюза. Шина наружу не торчит.

## 2. Контракты между модулями

> Контракты намеренно простые и стабильные. Имена полей — ориентировочные; финальные JSON-схемы — на фазе реализации с db-engineer. Важна **форма и направление**, не точные сигнатуры.

### 2.1. Клиент ↔ Шлюз (внешний контракт, WSS)
Двусторонний. Кадры идентифицируются полем `type`. Каждый кадр несёт `session_id`; сессия любого уровня дерева (координатор/под/проект) адресуется одним и тем же `session_id`. Для навигации UI событие/сессия дополнительно несут **путь в дереве** (`coordinator` | `pod_id` [+ `project_id`], см. §3.7) — сам контракт WSS/`seq`/докачки/красной зоны не меняется.

**Вверх (клиент → Шлюз):**
| Кадр | Назначение | Ключевые поля |
|---|---|---|
| `auth` | вход устройства | токен, идентификатор устройства |
| `subscribe` / `unsubscribe` | подписка окна на сессию | `session_id` |
| `message` | команда/реплика Владельца в чат | `session_id`, текст, (опц.) вложение |
| `confirmation.response` | ответ на «красную зону» | `confirmation_id`, `approved: true/false` |
| `interrupt` | прервать текущую работу ИИ | `session_id` |
| `resume_from` | докачка после обрыва | `session_id`, `last_seq` |

**Вниз (Шлюз → клиент):**
| Кадр | Назначение | Ключевые поля |
|---|---|---|
| `session.state` | снимок состояния сессии при подключении | статус, последний `seq`, есть ли ждущие подтверждения |
| `event` | событие действия ИИ (см. §2.3) | `session_id`, `seq`, событие |
| `message` | реплика ИИ | `session_id`, `seq`, текст |
| `confirmation.request` | запрос «красной зоны» | `confirmation_id`, описание действия, чем грозит |
| `error` | ошибка | код, человекочитаемое сообщение |

**Гарантия докачки:** каждый кадр вниз с `seq` сохраняется в журнале сессии; по `resume_from(last_seq)` Шлюз досылает всё с `last_seq+1`. У потоковых каналов «по умолчанию переигровки нет» — поэтому докачку закладываем с Фазы 1.

### 2.2. Шлюз ↔ Менеджер сессий (внутренний)
- `get_or_wake(session_id)` → дескриптор сессии (статус, поток событий, last_seq).
- `list_tree()` → иерархический реестр для Орбиты (Этаж 1): по каждому поду — его статус-сводка (= агрегат статусов проектов + собственной сессии пода, §3.7), и вложенный список проектов со своими статусами/тревогами/ждущими подтверждениями.
- `route_message(session_id, message)` → передать команду в chat-core сессии.
- `route_confirmation(confirmation_id, approved)` → передать ответ Владельца.
- `replay(session_id, from_seq)` → отдать пропущенные события из журнала.
- Шлюз подписывается на поток событий сессии (через event-bus) и перекладывает в WSS-кадры (§2.1).

### 2.3. Формат события действия ИИ (ядро контракта real-time)
Это **самый важный контракт продукта** — то, что делает «Мостик» не чат-ботом, а «руками в работе». Канонизируется адаптером (§1.3), стабилен независимо от версии SDK.

Каждое событие: `session_id`, `seq` (монотонный), `ts`, `type`, полезная нагрузка. Базовый словарь типов (MVP):

| `type` | Смысл (что показать Владельцу) | Нагрузка |
|---|---|---|
| `agent.started` | ИИ/субагент начал работу | имя, роль |
| `agent.thinking` | ИИ рассуждает (опц., можно сворачивать) | краткая сводка |
| `tool.call` | ИИ вызывает инструмент | инструмент, аргументы (с маскированием ПД/секретов) |
| `tool.result` | результат инструмента | статус, краткий вывод |
| `file.changed` | правка файла | путь, тип (create/edit/delete), diff-сводка |
| `confirmation.required` | действие красной зоны ждёт «да» | `confirmation_id`, описание, риск |
| `agent.finished` | работа/подзадача завершена | итог |
| `error` | сбой | код, сообщение |

Эти типы напрямую ложатся на «живой поток действий» Этажа 3 концепта (читаю файл X → запускаю агента → делаю правку → жду подтверждения). Расширять словарь — добавлением типов, не ломая существующие.

### 2.4. Менеджер сессий ↔ chat-core (внутренний)
- `spawn(node, env_type, perms)` / `resume(session_id)` / `suspend(session_id)` / `kill(session_id)`, где `node` несёт уровень и область видимости (координатор / под со списком путей проектов / проект с одним путём, §3.2).
- chat-core отдаёт наверх **канонический поток событий** (§2.3) с проставленными `seq`.
- принимает вниз: `message`, `confirmation_response`, `interrupt`.

### 2.5. chat-core ↔ файлы/агенты pod'а
- Рабочая директория жёстко = папка сессии (изоляция, §4.3).
- Инструменты: чтение/правка файлов, shell, git, запуск агентов с префиксом pod'а, деплой.
- «Красная зона» (§4.2) проходит через перехват адаптера до исполнения.

## 3. Модель сессий: ТРЁХУРОВНЕВОЕ ДЕРЕВО

> **Уточнение Владельца (msg 6844).** Сессии — не плоский список «одна на проект», а **дерево из трёх уровней**. Контракт real-time (§2: WSS, `seq`, докачка, красная зона) сохраняется как спроектирован; добавляется только адресация событий по пути «под → проект».

### 3.0. Три уровня дерева

```
УРОВЕНЬ 1  ┌──────────────────────────────────────────────┐
КООРДИНАТОР │  Координатор холдинга (обзор/межпроектный)     │  видит реестр всех подов,
ХОЛДИНГА    │  /root/  (реестр подов, без правки чужого)     │  раздаёт, не правит чужое
            └───────────────────────┬──────────────────────┘
                                    │
УРОВЕНЬ 2   ┌──────────┬────────────┼────────────┬──────────────┐
СЕССИЯ ПОДА │ ФБТО     │ НЗСН       │ М-ОС       │ … Дарвин(нов.) │  диалог по бизнесу в целом;
(направление)│         │            │            │                │  видит ВСЕ свои проекты,
            └──────────┴─────┬──────┴────────────┴──────────────┘  НЕ видит чужие поды
                             │
УРОВЕНЬ 3        ┌───────────┼───────────┐
СЕССИЯ ПРОЕКТА   │ «Сайт»    │ «ЦС НЗСН» │ «HR-отдел»   (внутри пода НЗСН)
                 └───────────┴───────────┘   видит только свою папку проекта
```

- **Уровень 1 — Координатор холдинга:** как и был. Обзор Орбиты (Этаж 1), межпроектные/стратегические задачи, маршрутизация. Не правит чужое без явного делегирования.
- **Уровень 2 — сессия ПОДА (направление):** ФБТО, НЗСН, М-ОС, Причал, Селижаровский карьер, Координата 56, ЖК Дарвин (новый). Диалог про бизнес-направление в целом. Область видимости — папки **всех проектов своего пода**, но НЕ чужих подов.
- **Уровень 3 — сессия ПРОЕКТА:** вложена в под. Пример: под НЗСН → «Сайт», «ЦС НЗСН», «HR-отдел». Область видимости — папка **конкретного проекта**.

Важно: **один под может содержать несколько репозиториев/папок проектов** — поэтому «под» это не одна папка `/root/<pod>/`, а *множество* путей проектов (см. §3.2 и config-as-data §3.6).

### 3.1. Жизненный цикл
Жизненный цикл (статусы, spawn/resume/suspend/kill) **одинаков для сессии любого уровня** — узел дерева есть узел дерева. Статусы в реестре Менеджера:

```
 (нет)──spawn──▶ ACTIVE ──suspend──▶ SUSPENDED ──resume──▶ ACTIVE
                   │  ▲                                       
                   │  └────────── wake (по message/resume_from) 
                   ▼
                WAITING_CONFIRMATION  (работа на паузе, ждёт «да» Владельца)
                   │ confirmation.response
                   ▼
                ACTIVE
                   │ kill / crash
                   ▼
                STOPPED ──resume(если state цел)──▶ ACTIVE
```

- **ACTIVE** — движок работает (думает/исполняет).
- **WAITING_CONFIRMATION** — упёрся в красную зону, работа заморожена, в очереди подтверждение.
- **SUSPENDED** — намеренно усыплён (экономия ресурсов), state на диске, будится по запросу.
- **STOPPED** — процесс остановлен/упал; при целом state — поднимается через `resume`.

### 3.2. Изоляция ПО УРОВНЯМ (область видимости — атрибут узла)
Изоляция теперь не «одна папка на сессию», а **scope (область видимости) зависит от уровня узла** в дереве:

| Уровень | Рабочая область (scope) | Видит | НЕ видит |
|---|---|---|---|
| **Координатор холдинга** | `/root/` — реестр подов (read) | сводку по всем подам | правку чужих файлов без явного делегирования |
| **Под (направление)** | СПИСОК путей всех своих проектов (может быть несколько репозиториев) | папки всех своих проектов | папки и агенты чужих подов |
| **Проект** | папка конкретного проекта | только свою папку | соседние проекты того же пода и чужие поды |

- Каждая сессия — свой процесс движка; рабочая область задаётся при `spawn` как **набор разрешённых путей** (для проекта — один, для пода — несколько, по числу репозиториев), а не один хардкод `/root/<pod>/`.
- Набор агентов сессии ограничен префиксом своего пода; кросс-проектные/кросс-подовые вызовы технически невозможны.
- История и контекст — отдельные у каждого узла дерева (history-store, §1.5).
- Профиль прав — отдельный (для MVP «Владелец = всё разрешено, кроме красной зоны без подтверждения»).
- **Это и безопасность** (взлом сессии проекта не открывает чужие проекты или поды), **и соблюдение** правила холдинга «один проект = свои агенты, кросс-вызовы запрещены».

### 3.3. Персистентность 24/7
Сессия живёт независимо от устройств. Закрыл ноутбук — движок продолжает (дописывает код, ждёт агента, готовит деплой).
- **Что персистится:** журнал событий с `seq`, история сообщений, очередь подтверждений, статус сессии, идентификатор движка для resume.
- **Где:** на диске VPS (history-store + state-файл сессии). Бэкап состояния — требование к инфра (§5, §6).
- **Требует проверки по офиц. документации:** насколько Agent SDK сам персистит контекст сессии и поддерживает `resume` между перезапусками процесса, либо контекст надо собирать самим из истории. От этого зависит, что именно мы храним.

### 3.4. Восстановление после обрыва КЛИЕНТА
Обрыв связи устройства ≠ остановка сессии. Сессия на VPS не замечает обрыва клиента — продолжает работу, события копятся в журнале с `seq`.

### 3.5. Докачка событий (replay)
При переподключении:
1. Клиент шлёт `auth` → `subscribe(session_id)` → `resume_from(last_seq)`.
2. Шлюз отдаёт `session.state` (текущий снимок) + досылает все `event`/`message` с `seq > last_seq` из журнала.
3. Ждущие подтверждения (`confirmation.request`) показываются заново — они **переживают обрыв** и не «зависают молча».
- Этот механизм — прямой ответ на риск «у потокового канала нет переигровки». Закладываем с Фазы 1, не откладываем.
- **Докачка работает поэлементно по `session_id`:** каждый узел дерева (под, проект, координатор) — отдельная сессия со своим журналом и своим `seq`. Подписка/докачка одинаковы независимо от уровня.

### 3.6. Дерево подов и проектов — config-as-data
Состав дерева **не зашит в код**, а описан данными (config-as-data), чтобы добавить новый под (ЖК Дарвин) или новый проект внутри пода было правкой конфигурации, а не релизом:
- **Под** = `pod_id`, имя, **список путей проектов** (один под — несколько репозиториев/папок), префикс агентов, профиль прав по умолчанию.
- **Проект** = `project_id`, родительский `pod_id`, путь(и) своей папки, профиль прав.
- Менеджер сессий читает эту конфигурацию и строит реестр-дерево; сессия конкретного узла создаётся лениво (по первому обращению), не все сразу.
- Стартовый список подов: ФБТО, НЗСН, М-ОС, Причал, Селижаровский карьер, Координата 56, ЖК Дарвин. Проекты внутри — по факту (пример НЗСН: «Сайт», «ЦС НЗСН», «HR-отдел»). **Точный состав путей/репозиториев каждого пода уточняется у Владельца/инфра** на фазе реализации.

### 3.7. Адресация событий по дереву
`session_id` каждого узла остаётся первичным адресом в контракте (§2). Дополнительно каждое событие/сессия несёт **путь в дереве** для навигации UI (Орбита → под → проект) и для маршрутизации:
- поля узла: `coordinator` | `pod_id` (+ `project_id` для уровня 3);
- Шлюз и UI используют путь, чтобы показать событие в правильном месте (на сфере пода, в кабинете проекта, в чате);
- `subscribe` может быть на узел любого уровня (Владелец говорит с подом НЗСН в целом или конкретно с проектом «Сайт»);
- агрегирование «здоровья» для Орбиты: статус пода = сводка статусов его проектов + собственной сессии пода.

## 4. «Руки» ИИ: безопасное исполнение и красная зона

### 4.1. Принцип
ИИ **читает/думает/готовит свободно**; **необратимое и опасное — только с явным «да» Владельца**. Реализуется перехватом на уровне адаптера (§1.3) ДО исполнения инструмента.

### 4.2. Красная зона (по умолчанию — максимально осторожный список)
Требуют `confirmation.request` → ждут ответа → исполняются только при `approved=true`:
- **деплой** (любой);
- **разрушительный git:** `push --force`, `reset --hard`, удаление веток/тегов;
- **удаление файлов/папок**, особенно вне рабочей зоны;
- **правка системных файлов** (`/etc`, сервисы);
- **любые «живые» внешние интеграции** (банки/Росреестр/1С-push/ОФД) — правило холдинга «no live external integrations», исключение только Telegram;
- **работа с секретами** (`.env`, `*.key`, `*credentials*`) — не логировать, не выводить, изменение — через подтверждение.

Зелёная зона (без спроса): чтение, рассуждение, правки файлов в своей рабочей папке, неразрушительный git (статус/diff/локальный коммит без push), запуск своих агентов на чтение/анализ.

**Список красной зоны — конфигурируемый** (config-as-data), по умолчанию осторожный. Окончательную границу утверждает Владелец (открытый вопрос §6).

### 4.3. Технические границы исполнения
- Рабочая директория сессии жёстко ограничена папкой pod'а.
- Маскирование ПД и секретов в событиях `tool.call`/`tool.result` и в аудите — **всегда**, с первой строки (правило холдинга).
- Канал только WSS; никакого открытого HTTP/WS.
- **Требует проверки:** механизм Agent SDK для (а) ограничения рабочей директории, (б) ограничения набора инструментов/команд, (в) перехвата вызова инструмента до исполнения. От этого зависит, делаем ли мы перехват средствами SDK или дополнительной обёрткой.

### 4.4. Аудит
Каждое: действие ИИ, команда Владельца, подтверждение/отклонение — в журнал аудита (что, когда, какой проект, по чьей команде, итог). ПД маскируется. Подход переиспользуем из M-OS (там аудит уже обязателен). Аудит — основа разбора инцидентов и будущего multi-user.

## 5. Декомпозиция на рабочие пакеты (backend-head → backend-dev)

> Без жёстких дедлайнов. Каждый пакет — самостоятельно проверяемый кусок (DoD-дисциплина). Не начинаем следующий, пока предыдущий не работает. **MVP = WP-0…WP-4** (закрывает Фазы 0–2 концепта на стороне бэкенда).

### MVP

**WP-0. Сверка возможностей Agent SDK (исследовательский, блокирующий).**
Перед любым кодом — `backend-dev` (или R&I) сверяет по **актуальной офиц. документации Agent SDK**: запуск как долгоживущего процесса, `resume`, перехват/одобрение вызова инструмента, ограничение рабочей директории и набора инструментов, формат потока событий. Результат — короткая записка «что SDK умеет из коробки, что добиваем обёрткой». От неё зависят WP-2 и WP-3.

**WP-1. Скелет Шлюза + одно WSS-соединение.**
Один endpoint WSS, базовая аутентификация (токен), приём `message`, отдача `message`/`event` вниз. Без мультиплексирования (одна сессия). Цель — труба от клиента до сервера.

**WP-2. Адаптер движка + одна сессия с руками.**
chat-core над Agent SDK, рабочая папка = `/root/coordinata56/`, канонизация событий (§2.3) в наш формат. Доказать end-to-end: «команда из чата → реальная правка файла на VPS → событие `file.changed` обратно в чат». Это Фаза 0 концепта.

**WP-3. Красная зона + перехват подтверждений.**
Перехват опасного инструмента → `confirmation.request` → пауза сессии (WAITING_CONFIRMATION) → ответ Владельца → исполнение/отмена. Список красной зоны как config-as-data (§4.2). Базовый аудит (§4.4).

**WP-4. Журнал событий с seq + докачка (replay).**
Персистентный журнал на сессию, монотонные `seq`, `resume_from(last_seq)`, `session.state` при подключении, переживающие обрыв подтверждения (§3.5). Закрывает Фазу 1 концепта на бэкенде. После этого бэкенд готов под Mac-клиент (Фаза 2 — это фронт+инфра, бэкенд уже их обслуживает).

### После MVP

**WP-5. Менеджер сессий: иерархическое дерево сессий (§3).**
Иерархический реестр (под → проекты), config-as-data состава подов/проектов (§3.6), жизненный цикл (spawn/resume/suspend/kill) для узла любого уровня, изоляция по уровням (scope = набор путей: проект — один, под — несколько репозиториев), мультиплексирование в Шлюзе (`subscribe`/`unsubscribe` на узел любого уровня, маршрутизация по `session_id` + путь дерева). `list_tree()` для Орбиты с агрегацией статуса пода. (Фаза 3.)

**WP-6. Сессия Координатора холдинга.**
Особый профиль (read реестра pod'ов, маршрутизация задач, без правки чужих файлов). Сводный обзор для Этажа 1. (Фаза 3.)

**WP-7. history-store как явный модуль + история для Этажа 2.**
Per-project история разговоров, выдача для Рабочего стола проекта. Согласование хранилища с db-engineer. (Фаза 5.)

**WP-8. Общее состояние Mac↔mobile + пуши.**
Fan-out на несколько окон одной сессии, события для пуш-уведомлений (ИИ закончил ночную работу / ждёт подтверждения). (Фаза 4.)

**WP-9. Закалка: event-bus под нагрузку, бэкапы state, задел гибрида.**
Вынос шины в лёгкий брокер при реальной многосессийной нагрузке, бэкапы состояния сессий, параметр «тип среды» = managed-Anthropic как опция. (Фаза 6.) Только при доказанной необходимости — против over-engineering.

**WP-10 (опц.). Multi-user RBAC на бэкенде.**
Профили прав per-user, проверка прав в Шлюзе и Менеджере. (Фаза 7, по решению Владельца.)

## 6. Зависимости и открытые вопросы

### 6.1. Зависимости от других департаментов
| От кого | Что нужно бэкенду |
|---|---|
| **Инфра/DevOps** | Где живёт VPS (общий/выделенный — §6.2 вопрос Владельцу); защищённое хранилище секретов и токенов; бэкапы состояния сессий; закрытые порты, WSS-only; мониторинг VPS как единой точки отказа |
| **Фронт** | Потребитель контракта §2.1 (WSS-кадры) и §2.3 (события действий); общее TS-ядро (клиент протокола, докачка, очередь подтверждений). Бэкенд фиксирует контракт — фронт реализует клиент |
| **QA** | Тест-сценарии: обрыв/докачка, переживающие обрыв подтверждения, изоляция сессий (сессия А не видит папку Б), красная зона не исполняется без «да», маскирование ПД/секретов в событиях и аудите |
| **db-engineer** | Согласование хранилища history-store и формата журнала событий/аудита на фазе реализации (MVP — простой append-файл, СУБД позже) |
| **Архитектор** | Утверждение этого брифинга на соответствие ADR-0001; разрешение спорных мест перехвата инструментов, если SDK не даёт нужного из коробки (WP-0) |

### 6.2. Открытые вопросы Владельцу (через Координатора)
1. **Граница красной зоны (§4.2):** утвердить список действий, требующих явного «да». По умолчанию предлагаю максимально осторожный (деплой, разрушительный git, удаление, внешние интеграции, секреты). Расширять/сужать?
2. **Где живёт VPS с «мозгом»:** текущий сервер холдинга или отдельный выделенный (изоляция критичной инфраструктуры + бюджет). Влияет на план инфра и бэкапов.
3. **Глубина MVP:** подтвердить, что первый срез — на ОДНОМ проекте (coordinata56), а не сразу 8 + Орбита. Это против красивого, в пользу работающего.

### 6.3. Технические места «требует проверки по офиц. документации» (сводно)
Зафиксированы для WP-0, чтобы не строить на памяти:
- запуск Agent SDK как долгоживущего серверного процесса;
- `resume` (возобновление сессии между перезапусками процесса) и что именно SDK персистит сам;
- перехват/одобрение вызова инструмента до исполнения (для красной зоны);
- ограничение рабочей директории и набора инструментов средствами SDK;
- формат потока событий движка (для канонизации в §2.3).
