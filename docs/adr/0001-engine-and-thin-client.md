# ADR-0001: Движок ИИ и модель «мозг на сервере + тонкие клиенты» для «Мостика»

- **Статус:** предложен (на согласование Владельца через Координатора)
- **Дата:** 2026-06-13
- **Контекст-документ:** `/root/m-orbital/docs/CONCEPT_bridge_owner_cockpit.md`
- **Решения, влияющие на бизнес:** да (§Последствия) — требуют подтверждения Владельца

---

## Контекст

Владелец заказал личный командный интерфейс для работы с ИИ — «Мостик»: нативное приложение в доке macOS + мобильное, все проекты холдинга открыты параллельно, внутри — чат с ИИ, обладающим **полной мощью** (правка файлов, запуск агентов, деплой), а не облегчённым чатом. Сессии должны жить на VPS 24/7 независимо от устройств; устройства — тонкие клиенты с общим состоянием.

Зафиксированные Владельцем решения (не пересматриваются): платформы Mac+mobile; одна сессия на проект со своей памятью/агентами + слой Координатора холдинга; уровень ИИ — настоящий Claude Agent SDK/Claude Code с инструментами; серверная жизнь сессий 24/7; v1 — один пользователь, архитектура multi-user-ready.

Два решения нужно зафиксировать архитектурно, потому что они определяют всё остальное:
1. **Чем реализовать движок ИИ с руками.**
2. **Где проходит граница между «думающим сервером» и «показывающим клиентом».**

---

## Проблема

«Полная мощь» означает реальный доступ к боевым папкам проектов (`/root/<pod>/`), их git, секретам, командам агентов и деплою. Нужно выбрать движок, который даёт этот доступ, не нарушая курс холдинга на полный контроль периметра (правила «no live external integrations», skeleton-first для ПД, маскирование/изоляция), и при этом совместим с уже работающей моделью «один pod = своя папка/агенты/CLAUDE.md».

---

## Решение 1 — Движок ИИ

### Рассмотренные альтернативы

**A. Claude Agent SDK / Claude Code как долгоживущие процессы на нашем VPS.**
ИИ с инструментами работает прямо на сервере Владельца, с родным доступом к папкам pod'ов, их агентам, git и деплою. Фактически — то, как холдинг управляется уже сейчас.

**B. Managed Agents Anthropic (`/v1/agents`, `/v1/sessions`, `/v1/environments`).**
Серверно-управляемые stateful-агенты; инструменты исполняются в контейнере **на инфраструктуре Anthropic**; файлы/репозитории монтируются; события идут потоком (SSE). Подтверждено по навыку claude-api: surface существует, beta-заголовок `managed-agents-2026-04-01`, окружения только `cloud`.

**C. Гибрид:** A как основа над боевыми pod'ами; B точечно как изолированная песочница для рискованных экспериментов и (в будущем) для урезанного доступа сотрудников.

### Выбор
**A — основной движок; C (гибрид) — заложенная опция через параметр сессии «тип среды» (local-VPS / managed-Anthropic).**

### Причины
1. Ценность продукта — руки в **реальных** папках; Managed Agents работают в чужом контейнере, куда пришлось бы копировать боевой код и секреты (сложнее + против контроля периметра).
2. Холдинг уже так управляется — «Мостик» даёт движку лицо и оркестровку, а не изобретает движок. Минимальная достаточность.
3. Managed Agents — beta; класть единственную опору флагмана на beta преждевременно.
4. Гибрид не теряется: «тип среды» — параметр сессии, B подключаем точечно позже (песочница, multi-user).

### Не делаем
- Не строим основу на Managed Agents.
- Не тянем боевые секреты в сторонние контейнеры.

## Решение 2 — «Мозг на сервере + тонкие клиенты»

### Рассмотренные альтернативы

**A. Толстый клиент:** движок и логика — на устройстве; сервер — вспомогательный. Отвергнут: ломает требование «сессии живут 24/7 независимо от устройств» и единое состояние Mac↔mobile; на телефоне исполнять руки ИИ небезопасно и тяжело.

**B. Тонкий клиент, мозг на сервере (выбран):** все сессии и инструменты на VPS; клиенты — окна, получают события и шлют команды по WSS; состояние — на сервере.

**C. Бессерверная синхронизация (peer-to-peer между устройствами):** отвергнут — нет единой 24/7-точки, негде безопасно держать руки ИИ и аудит, сложная согласованность.

### Выбор
**B — тонкий клиент, мозг на сервере.** Шлюз (gateway) ⟶ менеджер сессий ⟶ движок сессии (chat-core). Реальное время — WSS как основной канал (SSE — запасной для односторонней выдачи). Состояние и журнал событий с порядковыми номерами — на сервере; при обрыве клиент догружает пропущенное по последнему номеру; запросы подтверждений переживают обрыв.

### Причины
Прямо реализует зафиксированные Владельцем решения (24/7, общее состояние, тонкие клиенты), даёт единую точку аудита и контроля рук ИИ, переиспользует существующую Hub/pod-модель.

## Последствия

**Положительные:**
- Полный контроль периметра; руки ИИ — на реальных проектах.
- Переиспользование текущей pod-инфраструктуры и активов (`m-orbital`, «Кардинал»).
- Multi-user и гибридная песочница достижимы наращиванием, без переписывания.

**Отрицательные / издержки:**
- VPS — единая точка отказа (нужны бэкапы состояния, мониторинг, в перспективе резерв).
- Реальное время требует аккуратной реализации (догрузка событий, переживающие обрыв подтверждения).
- Часть деталей Agent SDK нельзя брать по памяти — перед реализацией Фазы 0 обязательна сверка с актуальной официальной документацией Agent SDK.

**Бизнес-последствия (требуют подтверждения Владельца):**
- Выбор движка A над B; гибрид — на потом.
- Где размещать VPS с «мозгом» (общий сервер холдинга или выделенный — вопрос изоляции критичной инфраструктуры и бюджета).
- Граница «красной зоны» действий, требующих явного подтверждения.

---

## Риски

| Риск | Смягчение |
|---|---|
| Руки ИИ на боевых папках (ошибка/взлом дорого стоят) | Подтверждение опасных действий, изоляция сессий по папке+агентам, аудит, отдельная фаза безопасности перед расширением доступа |
| VPS — единая точка отказа | Бэкапы состояния сессий, мониторинг, резервирование в перспективе |
| Beta-зрелость Managed Agents | Основа = A (GA); B опционально |
| Неточность возможностей Agent SDK по памяти | Сверка с офиц. документацией перед Фазой 0; в ADR фиксируется архитектурная роль, не сигнатуры |
| Over-engineering из-за масштаба амбиции | Жёсткая фазовость, MVP = узкий вертикальный срез на одном проекте |
