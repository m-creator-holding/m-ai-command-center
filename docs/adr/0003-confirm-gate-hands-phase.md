# ADR-0003: Контур подтверждения опасных действий («руки с подтверждением»)

- **Статус:** предложен (на согласование Владельца через Координатора)
- **Дата:** 2026-06-13
- **Контекст-документы:** `adr/0001-engine-and-thin-client.md`, `adr/0002-session-os-isolation.md`, `dept/infrastructure.md` (§5 аудит/красная зона), `dept/quality.md`, код `command-center-engine/`
- **Решения, влияющие на бизнес:** да — определяет, как Владелец санкционирует правки/деплой

---

## Контекст

Продукт обещает Альфреда «с руками»: он не только читает, но и правит файлы и публикует. Ключевое требование Владельца — **красная зона**: опасное действие выполняется только после явного подтверждения человеком. ADR-0001 фиксирует поэтапность read → write → deploy, каждый этап — отдельный gate.

К этому моменту построен фундамент, общий для любого механизма подтверждения (коммит движка `05a69ce` и следующий):

- **Профили прав** (`engine/profiles.py`): `read-only` (Фаза 0) и `write` (Edit/Write/NotebookEdit, scope на запись; Bash остаётся запрещён до профиля `deploy`). Неизвестный профиль → read-only (fail-safe).
- **ОС-изоляция** (`engine/sandbox.py`, ADR-0002): scope монтируется на запись только для write-профиля; чужие поды физически недостижимы.
- **path_guard на запись** (`gateway/path_guard.py`): Write/Edit по абсолютному пути вне scope отклоняется (раньше стерёг только чтение). Покрыто тестами (`..`, симлинк-побег, соседний префикс).
- **Красная зона** (`engine/redzone.py`): `check()` уже классифицирует write/bash/неизвестное → `confirm` (fail-safe), `mask()` прячет секреты.
- **Журнал аудита** (`engine/audit.py`): append-only JSONL на проект, маскированный, fail-safe.

Не хватает **самого механизма подтверждения**: при raw-headless `claude -p` процесс исполняет инструменты внутри себя — нет естественной точки «остановись, спроси человека, продолжи».

---

## Проблема

Дать Владельцу санкционировать каждое опасное действие движка до его исполнения, сохранив:

- безопасность (ничего необратимого без явного «да»);
- понятность (Владелец видит, ЧТО именно будет сделано, простым языком);
- аудит (любое действие и решение — в журнал);
- совместимость с уже работающим headless-движком и тонким клиентом (WebSocket).

---

## Рассмотренные альтернативы

**A. Предложение → подтверждение → применение движком (deterministic apply).**
Альфред работает в read-only, ПРЕДЛАГАЕТ изменение как структурированную правку (какой файл, какой фрагмент → на что). Интерфейс показывает карточку (ConfirmCard уже есть во фронте). Владелец подтверждает. Изменение применяет наш детерминированный код внутри writable-песочницы — БЕЗ обращения к ИИ (стоимость $0 на шаг применения).
*Плюсы:* максимум контроля, claude вообще не получает «сырые» инструменты записи; шаг применения дешёвый, повторяемый, полностью логируемый. *Минусы:* ограничено правками, выразимыми как патч; многошаговые сложные изменения неудобны.

**B. Мост через permission-prompt-tool (нативный механизм Claude Code).**
`claude` запускается с инструментами записи и флагом `--permission-prompt-tool`, указывающим на наш MCP-инструмент. Когда Альфред хочет записать — claude вызывает наш инструмент, тот пересылает запрос подтверждения по WebSocket Владельцу, ждёт «да/нет» и возвращает allow/deny; claude продолжает или отказывается.
*Плюсы:* настоящая агентная правка с пошаговым подтверждением — ровно дух продукта; родной механизм. *Минусы:* больше движущихся частей (MCP-сервер прав), процесс claude живёт дольше (ждёт ответа человека), для проверки нужен живой claude (упирается в бюджет).

**C. Plan-mode + acceptEdits.**
Альфред в режиме плана предлагает план целиком; Владелец одобряет план; повторный прогон с `acceptEdits` исполняет автономно внутри scope.
*Минусы:* грубое «всё или ничего», `acceptEdits` снимает пошаговый контроль — против духа красной зоны.

---

## Решение

**Целевой механизм — B (permission-prompt-tool): настоящие «руки с подтверждением».** Это соответствует видению продукта (Альфред действует, Владелец санкционирует опасные шаги пошагово).

**Первый поставляемый срез записи — A (предложение → подтверждение → применение).** Самый низкий риск, доказывает петлю «команда → правка → результат» дёшево и полностью под контролем; идеален для первого боевого примера (например, «поправь телефон в контактах сайта»). claude при этом не получает сырых инструментов записи вообще.

Путь: сначала A (узкий, безопасный, дешёвый), затем B (общая агентная правка) поверх того же фундамента. Профиль `deploy` (публикация, git) — следующий gate после устойчивого write.

Фундамент (профили, песочница, path_guard на запись, красная зона, аудит) — **механизм-независимый**: годится и для A, и для B. Поэтому он построен и протестирован сейчас, до выбора Владельцем механизма и до живого теста.

---

## Что доказано сейчас (без обращения к API, $0)

`pytest` (32 теста, проходят) + `ruff`/`mypy` чисто:

- read внутри scope — ок; read/write/edit вне scope — отклонены (`..`, симлинк-побег, соседний префикс);
- write-профиль включает Edit/Write, но НЕ Bash; read-only исключает запись; неизвестный профиль → read-only;
- write-инструмент внутри scope канонизируется в `tool.call` с `decision=confirm`; вне scope — бросает path_guard;
- красная зона: read→allow, write/bash/неизвестное→confirm; маскирование .env/Bearer/секретных файлов;
- журнал аудита: append-only, разделение по сессиям, fail-safe.

ОС-изоляция (ADR-0002) подтверждена отдельно (`tools/test_sandbox.sh`).

## Последствия

**Положительные:**
- Фаза записи строится безопасно и поэтапно; каждый кусок проверяем при $0.
- Включение записи на боевом поде остаётся отдельным gate Владельца — фундамент по умолчанию выключен (профиль пода = `read-only`, песочница off).
- Аудит и path_guard на запись закрывают разрушительные сценарии до того, как руки реально включат.

**Издержки / ограничения:**
- Живой тест любой из веток (A применяет патч; B — claude пишет под подтверждением) ещё не прогнан — упирается в решение по бюджету движка и (для B) в MCP permission-сервер.
- Профиль `deploy` (git/публикация) намеренно не трогаем: shell под запретом во всех профилях, кроме будущего `deploy`.

## Открытые вопросы Владельцу (через Координатора)

1. **Какой механизм первым:** согласовать «сначала A (Альфред предлагает — Вы жмёте «применить») как первый безопасный срез, затем B (свободная правка с пошаговым подтверждением)»?
2. **Первый сценарий записи** для боевого среза — предложите простой реальный кейс (кандидат: «поправить телефон в контактах Сайта НЗСН»), чтобы доказать петлю на безопасном изменении.
3. **Связка с бюджетом:** живой write-тест тратит ИИ — запускать после решения подписка vs ключ (ADR-0002 §3) или разово на копейки для доказательства петли?
