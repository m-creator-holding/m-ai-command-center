# ADR-0002: ОС-изоляция сессий движка (песочница перед фазой «рук»)

- **Статус:** предложен (на согласование Владельца через Координатора)
- **Дата:** 2026-06-13
- **Контекст-документы:** `adr/0001-engine-and-thin-client.md`, `dept/infrastructure.md` (§2 «Изоляция сессий»), код движка `command-center-engine/`
- **Решения, влияющие на бизнес:** да (§Последствия и §Открытые вопросы Владельцу)

---

## Контекст

Движок Фазы 0 (`/root/command-center-engine`) работает: «Альфред» читает «Сайт НЗСН» headless через `claude -p ... --output-format stream-json`, поток событий идёт в интерфейс. Текущая защита границ — два уровня **приложения**:

- флаги запуска (`--allowedTools Read Grep Glob --disallowedTools Bash Edit Write`) — режим только чтения;
- `gateway/path_guard.py` — пост-проверка путей в аргументах инструментов: вызов вне scope → событие `error` + остановка сессии.

**Находка Т4 (наблюдаемая, не гипотеза):** рабочий каталог (`cwd = scope`) НЕ ограничивает чтение файлов. Read с абсолютным путём спокойно выходит за scope — read-only сессия «Сайта НЗСН» физически прочитала `/root/prichal-cafe` (чужой под). `path_guard` ловит это на стороне gateway, но сам процесс `claude` под root по-прежнему имеет доступ ко всей файловой системе.

Перед тем как давать движку «руки» (write → deploy, ADR-0001), прикладной защиты недостаточно: ошибка/обход на уровне приложения = доступ ко всем боевым папкам, секретам и БД холдинга. Нужен **барьер операционной системы**, а не только проверка в коде.

---

## Проблема

Дать сессии проекта доступ ровно к своей папке и ничему больше — на уровне ядра, а не на доверии к коду, — но:

- не ломая уже работающий read-only поток Фазы 0;
- не запуская сейчас чувствительную миграцию root-модели (отъём root у всего холдинга);
- сохранив возможность `claude` авторизоваться и ходить к API Anthropic;
- подготовив, но не включив, write-режим будущей фазы «рук».

---

## Рассмотренные альтернативы

**A. Отдельный системный пользователь на сессию (`bridge-<pod>`) + права ФС.**
Уровень 1 из инфраструктурного брифинга. Надёжно, но требует отъёма root и смены владельца боевых папок `/root/<pod>/` — самый чувствительный пакет (DEVOPS-2), трогает текущую модель всего холдинга, нужен план отката. Это решение и миграция — **вопрос Владельцу**, не делается автономно.

**B. Mount-namespace через bubblewrap (`bwrap`).**
Уровень 2 из брифинга. Сессия видит изолированное представление ФС: примонтирован только её scope + минимум системных каталогов; `/root` и чужие поды в namespace **физически не существуют**. Не требует отъёма root у холдинга и смены владельца боевых папок — изоляция даётся на запуске процесса. `bwrap` уже установлен на VPS (проверено), ядро 6.8, user-namespaces включены.

**C. Контейнер (Docker) на сессию.**
Тяжелее, дольше старт, свой жизненный цикл образов; на общем VPS рядом с 13 боевыми контейнерами добавляет нагрузку. Избыточно для Фазы 0; разумно на фазе закалки/выделенном сервере.

---

## Решение

**Принять B (bubblewrap) как механизм ОС-изоляции сессий движка, начиная с Фазы 0.**

Реализовано в `engine/sandbox.py`. Команда `claude` оборачивается в `bwrap` так, что внутри namespace видны ТОЛЬКО:

- `/usr`, `/bin`, `/lib`, `/lib64`, `/etc` — read-only (бинарь `claude` + node-рантайм + TLS-сертификаты);
- папка проекта (scope) → `/work` (**read-only в Фазе 0**; `writable=True` подготовлен для фазы «рук»);
- синтетический HOME: **только** файл авторизации `/root/.claude/.credentials.json`, смонтированный read-only. Всё остальное из `/root/.claude` (`secrets/`, чужие проекты, агенты, хуки, история других сессий) в namespace отсутствует.

`/root` целиком в песочнице не существует. Namespaces: user/pid/ipc/uts изолированы, `--die-with-parent`, `--new-session`. **Сеть не изолируется** (хостовая) — `claude` должен достигать API Anthropic; ограничение сети только до Anthropic — отдельная закалка.

Соотношение с `path_guard`: оба остаются. `path_guard` — быстрый прикладной отказ с понятным сообщением пользователю; песочница — финальный барьер ОС («защита в глубину»). Если код обойдут — ядро всё равно не покажет чужой под.

**Включение — опциональное, через `CC_ENGINE_SANDBOX=1`.** По умолчанию выключено: проверенный read-only поток Фазы 0 не меняется до живого smoke-теста под API (висит на решении по бюджету движка).

---

## Что доказано (не память — прогон)

`tools/test_sandbox.sh` (стоимость $0, без обращения к API):

1. scope виден как `/work` — читается;
2. `/root` в namespace отсутствует;
3. чужой под `/root/prichal-cafe` недостижим — **находка Т4 закрыта ОС-барьером**;
4. `/root/.claude/secrets` недостижим;
5. `claude --version` запускается внутри песочницы (бинарь+node+TLS на месте).

Все пять проверок проходят. `ruff`/`mypy` по `engine/` — чисто.

## Последствия

**Положительные:**
- ОС-барьер закрывает находку Т4 без отъёма root у холдинга и без смены владельца боевых папок.
- «Защита в глубину»: прикладной `path_guard` + ОС-изоляция.
- Подготовлен write-режим (`writable=True`) — фаза «рук» получает изоляцию сразу, не дописываясь поверх голого root.
- Нулевой бюджет на инфраструктуру (`bwrap` уже стоит).

**Издержки / ограничения:**
- Сеть пока хостовая — сессия технически достигает localhost-портов боевых сервисов. Ограничение сети до Anthropic — следующая закалка (firewall/slirp/сетевой namespace с whitelist).
- Песочница bwrap изолирует ФС/процессы, но процесс всё ещё стартует от root-пользователя хоста (user-namespace маппит, но это не полноценный непривилегированный пользователь). Для write-фазы рекомендуется СОВМЕСТИТЬ B с A (отдельный пользователь) — тогда даже при побеге из namespace права ФС вторым рубежом.
- Живой smoke-тест «claude делает реальный API-вызов внутри песочницы» ещё не прогнан (бережём бюджет движка); ожидаемо требует лишь `/etc/resolv.conf` + CA-сертификаты, которые уже в `--ro-bind /etc`. Оценка стоимости — копейки (короткий read-only промпт), не 68k-токенный полный обзор сайта.

## Открытые вопросы Владельцу (через Координатора)

1. **Совмещать ли B с A для фазы «рук»?** Рекомендация Директора инфраструктуры — да (двойной рубеж). Это и есть чувствительная миграция root-модели (DEVOPS-2): согласны переводить сессии движка с root на служебных пользователей `bridge-<pod>` перед включением записи/деплоя?
2. **Ограничение сети сессии** до API Anthropic + git-remote своего пода (отрезать localhost с чужими БД) — закладывать в эту же фазу безопасности или на фазе закалки?
3. **Живой smoke-тест песочницы под API** (копейки) — разрешить разовый прогон, чтобы подтвердить реальный ответ Альфреда из песочницы, или ждать общего решения по бюджету движка?
